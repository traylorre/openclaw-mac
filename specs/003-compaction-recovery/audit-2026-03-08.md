## Weak Spots

### W-001: T007 is dangerously overloaded

T007 (compaction-audit --batch) carries ~20 revert-related FRs (036-042, 048-049,
057, 059, 061, 066-067, 096-100) but only cites 5. An implementer following T007
alone would miss critical edge cases.

**Fix**: Split T007 into sub-tasks: (a) core batch revert loop, (b) edge-case
handling (conflicts, type changes, shallow clone, index.lock), (c) recovery log
writing + sentinel creation.

### W-002: No abort mechanism (FR-032)

Zero coverage for "developer wants to cancel recovery." No Ctrl+C cleanup path,
no abort command. Only escape is manually deleting `.claude/recovery-marker.json`.

**Fix**: Add a task for abort handling, or explicitly document manual escape hatch.

### W-003: No resume-from-interruption (FR-063)

Marker tracks `stage` but no task implements "marker exists at stage X from a
crashed prior recovery — what now?" T006 handles staleness (different session)
but not resumption (same session, interrupted).

**Fix**: Add resume logic to T006 or a new task.

### W-004: Testing is dry-run only

No mock transcripts, no simulated compaction events, no integration test harness.
T018 pipes JSON through dry-run mode but can't test actual hook flow, watcher
polling, or tmux integration.

**Fix**: Accept this for MVP, but note the risk.

### W-005: CARRYOVER suppression (FR-017) is implicit

The hook-compact contract lists "suppresses CARRYOVER auto-clear via marker
presence" as side-effect #5, but no task describes the integration point: feature
002's loader must check for the marker and skip.

**Fix**: Add a note to T006 or T012, or defer until feature 002 exists.

### W-006: Infrastructure file ordering (FR-048)

T007 mentions "flag infrastructure files" but doesn't specify that their revert
must complete before sentinel creation. Sequencing is implicitly correct (all
reverts finish before sentinel), but priority ordering within the revert loop
isn't called out.

**Fix**: Add explicit ordering note to T007.
