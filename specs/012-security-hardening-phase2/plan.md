## Summary

Comprehensive security hardening building on 011-workspace-integrity. Expands protection from 49 files to 65+, adds hash-chained audit logging, container integrity verification, browser session encryption, webhook payload sanitization, manifest rollback detection, and audit enforcement gates. Grounded in adversarial review (18 findings), ecosystem research (ClawHavoc: 1,184 malicious skills, 31 CVEs), and NIST/OWASP/MITRE frameworks.

## Implementation Phases

### Phase 1: Expanded Protection Surface (US1)

**Purpose**: Add all newly identified files to the protected set, sign the allowlist, clean up git hooks and restore scripts.

**Tasks**:
1. Expand `_integrity_protected_file_patterns()` in `lib/integrity.sh` with R-007 file list
2. Add HMAC signing to `skill-allowlist.sh` (wrap in state-file signing pattern)
3. Update `integrity-verify.sh` to verify allowlist signature (FR-008)
4. Add git hooks neutralization to `integrity-deploy.sh` (chmod -x, lock)
5. Add restore scripts permission tightening to `integrity-deploy.sh`
6. Add old backup detection and locking to `integrity-lock.sh`
7. Add TMPDIR validation to `integrity_check_env_vars()` (FR-035)
8. Add lib self-verification to all new scripts (FR-034)
9. Verification: deploy + lock → verify expanded manifest includes new files

**Dependencies**: None (first phase)
**Risk**: Protected file list growth increases deploy/lock time. Mitigate by parallelizing checksums.

### Phase 2: Hash-Chained Audit Log (US2)

**Purpose**: Implement tamper-evident audit logging with append-only enforcement.

**Tasks**:
1. Update `integrity_audit_log()` in `lib/integrity.sh` to include `prev_hash` field
2. Add hash chain verification function `integrity_verify_audit_chain()`
3. Add `chflags uappnd` setup to `integrity-deploy.sh` (run after first log entry)
4. Update all operations (lock, unlock, deploy, verify, allowlist, monitor) to log through the audit function
5. Add audit log verification to `integrity-verify.sh`
6. Add CHK-OPENCLAW-AUDIT-CHAIN audit check to `hardening-audit.sh`
7. Verification: append entries → verify chain → attempt truncation → verify denied

**Dependencies**: Phase 1 (expanded file list must include audit log path)
**Risk**: `uappnd` prevents log rotation. Mitigate by documenting rotation procedure (requires sudo to clear flag temporarily).

### Phase 3: Container and Docker Integrity (US3)

**Purpose**: Verify n8n container image and credential set.

**Tasks**:
1. Add container image ID capture to `integrity-deploy.sh` (R-004)
2. Add container image verification to `integrity-verify.sh` (FR-015)
3. Add credential enumeration to `integrity-verify.sh` (FR-017)
4. Add container image monitoring to `integrity-monitor.sh` heartbeat cycle
5. Add CHK-OPENCLAW-CONTAINER-IMAGE audit check
6. Add CHK-OPENCLAW-CONTAINER-CREDS audit check
7. Verification: record image → replace container → verify detection

**Dependencies**: Phase 1 (manifest extension with container fields)
**Risk**: `docker inspect` requires Docker socket access. Container may be stopped during check. Mitigate with graceful skip + warning.

### Phase 4: Browser Session Encryption (US4)

**Purpose**: Encrypt LinkedIn storageState at rest.

**Tasks**:
1. Create `scripts/session-encrypt.sh` with encrypt/decrypt subcommands (R-003)
2. Generate and store encryption key in Keychain (separate from HMAC key)
3. Add session decrypt audit logging (FR-021)
4. Add secure deletion of plaintext after encryption (FR-022)
5. Add CHK-OPENCLAW-SESSION-ENCRYPTED audit check
6. Verification: encrypt → verify not plaintext → decrypt → verify usable

**Dependencies**: None (independent of other phases)
**Risk**: If Keychain entry is lost, session cannot be decrypted. Mitigate by documenting recovery (re-login to LinkedIn manually).

### Phase 5: Output Sanitization (US5)

**Purpose**: Validate webhook payloads before they reach external APIs.

**Tasks**:
1. Add sanitization Code node to `workflows/linkedin-post.json` after HMAC verify
2. Implement schema validation (required fields, types, no unexpected fields)
3. Implement content sanitization (control characters, length limits)
4. Add sanitization rejection logging
5. Add linkedin-post.json to protected file list (FR-036)
6. Verification: send injection payload → verify rejection → send valid payload → verify pass

**Dependencies**: Phase 1 (protected file list for workflow files)
**Risk**: Sanitization in n8n is in the same trust zone as the container (circular dependency per adversarial review FINDING-04). Mitigate by documenting this and protecting the workflow file.

### Phase 6: Manifest Versioning (US6)

**Purpose**: Detect manifest rollback attacks.

**Tasks**:
1. Add `manifest_sequence` to manifest build in `lib/integrity.sh` (FR-027)
2. Create `~/.openclaw/manifest-sequence.json` signed state file (R-006)
3. Add sequence verification to `integrity-verify.sh` (FR-028)
4. Add `--force` flag to `integrity-deploy.sh` for intentional resets (FR-030)
5. Verification: deploy v1 → deploy v2 → replace with v1 → verify rollback detected

**Dependencies**: Phase 2 (audit logging must be operational for rollback event logging)
**Risk**: Sequence file adds another signed artifact. Mitigate by using existing `integrity_sign_state_file()`.

### Phase 7: Audit Enforcement Gate (US7)

**Purpose**: Critical audit checks block agent launch.

**Tasks**:
1. Create `scripts/enforcement-setup.sh` for configuration management (R-008)
2. Create `~/.openclaw/enforcement.json` signed configuration
3. Modify `integrity-verify.sh` to load enforcement config and fail (not warn) on enforced checks
4. Add `FORCE=1` bypass with audit logging (FR-032)
5. Add enforcement.json to protected file list
6. Add CHK-OPENCLAW-ENFORCEMENT audit check
7. Verification: disable sandbox → verify blocked → enable → verify pass → FORCE=1 → verify bypass logged

**Dependencies**: Phase 1 (file in protected set), Phase 2 (audit log for bypass logging)
**Risk**: Over-enforcement could lock out the operator during debugging. Mitigate with `FORCE=1` bypass and hardcoded minimum set.

## Research Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| R-001 | `chflags uappnd` for audit log | Root can clear (accepted risk); `sappnd` too operationally hostile |
| R-002 | Hash-chained audit entries | Detects insertion/reordering; combined with uappnd prevents deletion |
| R-003 | AES-256-GCM via openssl for session encryption | Built-in, no deps; key in separate Keychain entry |
| R-004 | Docker image ID in manifest | Content-addressed, definitive; cheap to verify |
| R-005 | Sanitization inside n8n (Code node) | Simpler than proxy; circular dependency mitigated by protecting workflow file |
| R-006 | Separate signed sequence file | Prevents manifest+sequence rollback in single operation |
| R-007 | 8 new file categories in protected list | Each identified in adversarial file inventory |
| R-008 | Signed enforcement config with hardcoded minimum | Configurable maturity path; minimum prevents total disable |

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Trust anchor compromise (ADV-001) | MEDIUM | CRITICAL | Documented as known limitation; Keychain hardening deferred to M5 |
| Audit log fills filesystem | LOW | MEDIUM | Document rotation procedure; monitor disk usage |
| Container stopped during check | MEDIUM | LOW | Graceful skip with warning |
| Operator locked out by enforcement | LOW | HIGH | FORCE=1 bypass; hardcoded minimum is small |
| Over-engineering for single-operator setup | MEDIUM | MEDIUM | Scope bounded to 7 user stories; M5 for enterprise features |
| Deploy time increases with more protected files | LOW | LOW | Parallelize checksums; ~65 files is manageable |
| Key rotation affects 7 signed artifacts | LOW | HIGH | Document rotation procedure; test in verification |
