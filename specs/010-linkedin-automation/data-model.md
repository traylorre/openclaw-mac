## Entity: Content Draft

A piece of content generated by the agent, pending operator review.

**Attributes**:
- `id`: Unique identifier (UUID)
- `type`: `post` | `image_post` | `article_share`
- `content`: Draft text content
- `image_path`: Path to image file (for image posts, null otherwise)
- `status`: `drafted` | `presented` | `approved` | `rejected` | `published` | `failed`
- `created_at`: Timestamp of draft creation
- `presented_at`: Timestamp when shown to operator
- `resolved_at`: Timestamp of approval/rejection/publication
- `revision_count`: Number of edit cycles

**State Transitions**:
```
drafted → presented → approved → published
                   → rejected → (discarded)
                   → approved → failed → (retry or alert)
         presented → edited → presented (loop)
```

**Persistence**: Stored in `~/.openclaw/agents/<agentId>/pending-drafts.json` for restart recovery. Cleared after resolution (approved+published, rejected, or failed+acknowledged).

**Relationships**:
- Belongs to one Persona Configuration (inherits voice/rules)
- Produces one Activity Record when published

## Entity: Activity Record

A log entry for any action taken by the system.

**Attributes**:
- `execution_id`: n8n execution ID (primary key in execution history)
- `workflow_id`: Which n8n workflow was executed
- `workflow_name`: Human-readable workflow name
- `action_type`: `post` | `comment` | `like` | `alert` | `token_check`
- `status`: `success` | `failed` | `cancelled`
- `started_at`: Execution start timestamp
- `completed_at`: Execution end timestamp
- `input_summary`: Key input data (post topic, target URN, etc.)
- `output_summary`: Key output data (LinkedIn post URL, error message, etc.)

**Persistence**: Stored in n8n execution history (n8n database, Docker volume). Queryable via n8n REST API (`GET /api/v1/executions`). Retained according to n8n's execution data retention settings.

**Relationships**:
- Links to the n8n workflow that produced it
- May reference a Content Draft (for post/comment actions)

## Entity: Persona Configuration

The set of files defining the agent's voice, knowledge, and operating rules.

**Attributes**:
- `agent_id`: OpenClaw agent identifier
- `workspace_path`: `~/.openclaw/agents/<agentId>/agent/`
- `files`:
  - `SOUL.md`: Persona definition (tone, values, behavioral boundaries)
  - `AGENTS.md`: Operating rules (posting frequency, content boundaries, approval requirements)
  - `USER.md`: Operator context (who they are, communication preferences)
  - `TOOLS.md`: Tool documentation (informational, does not grant permissions)
  - `IDENTITY.md`: Agent name, character designation
  - `BOOT.md`: Startup sequence (restart recovery, pending draft re-prompting)
- `checksums`: SHA-256 hash of each file, stored in `~/.openclaw/manifest.json`

**Integrity verification**: The hardening audit computes current SHA-256 checksums and compares against manifest values. Mismatches flagged as WARN (operator may have intentionally edited files).

**Relationships**:
- Controls Content Draft generation (voice, rules, boundaries)
- Referenced by hardening audit checks (CHK-OPENCLAW-WORKSPACE-*)

## Entity: Credential Lifecycle State

The current status of the LinkedIn API credential with dual-token support.

**Attributes**:
- `access_token_granted_at`: When the access token was last granted or refreshed (stored in n8n Workflow Static Data)
- `access_token_expiry_days`: 60 (LinkedIn standard for consumer OAuth access tokens)
- `access_token_days_remaining`: `(access_token_granted_at + 60 days) - now()`
- `refresh_token_granted_at`: When the refresh token was originally issued (stored in n8n Workflow Static Data)
- `refresh_token_expiry_days`: 365 (LinkedIn refresh token TTL, does not reset on use)
- `refresh_token_days_remaining`: `(refresh_token_granted_at + 365 days) - now()`
- `last_refresh_attempt`: Timestamp of last automated refresh attempt
- `last_refresh_result`: `success` | `failed` | `not_attempted`
- `refresh_retry_count`: Number of consecutive refresh failures (reset on success, max 3 for server errors)
- `refresh_in_progress`: Boolean flag preventing concurrent refresh attempts
- `refresh_token_expired`: Boolean circuit breaker — when true, skips automated refresh
- `status`: `healthy` | `expiring_soon` (access ≤7 days) | `refresh_expiring_soon` (refresh ≤30 days) | `expired` | `refresh_token_expired` | `refresh_failed`

**Persistence**: All timestamps and flags stored in n8n Workflow Static Data (persists across workflow executions in production mode).

**Token lifecycle**:
1. Operator completes OAuth authorization (manual browser flow)
2. `access_token_granted_at` and `refresh_token_granted_at` recorded (same timestamp on initial grant)
3. Daily check computes days remaining for both tokens
4. When access token ≤7 days: automated refresh via LinkedIn OAuth endpoint
5. On successful refresh: `access_token_granted_at` updated, 60-day window resets
6. Refresh token TTL does NOT reset on use (365 days from original issue)
7. When refresh token ≤30 days: alert operator to plan re-authorization
8. When refresh token expired: circuit breaker set, alert operator for full manual re-auth

**Security note (ASI04)**: The 365-day refresh token increases the blast radius of a credential compromise vs. the previous 60-day access-only model. Mitigated by n8n credential isolation (agent never holds tokens directly) and Docker container boundaries.

**Relationships**:
- Monitored by the token-check scheduled workflow
- Blocks Content Draft publishing when expired
- Produces Activity Records for token checks, refresh attempts, and alerts
- References 014 Token Lifecycle State entity for the hardening perspective

## Entity: Operating Configuration

Operator-defined parameters controlling system behavior.

**Active Attributes** (US1):
- `active_hours_start`: Hour of day (e.g., 8)
- `active_hours_end`: Hour of day (e.g., 22)
- `quiet_hours_start`: Hour of day (e.g., 22)
- `quiet_hours_end`: Hour of day (e.g., 7)
- `daily_post_limit`: Max posts per day (default: 1-3)
- `timing_randomization_range_minutes`: Min/max delay between actions (e.g., 5-30)

**Future Attributes** (deferred to US2):
- `mode`: `warmup` | `steady_state`
- `daily_comment_limit`: Max comments per day
- `daily_like_limit`: Max likes per day

**Persistence**: Stored as n8n Custom Variables, accessible via `$vars.<name>` in workflow expressions and updatable via n8n REST API or UI.

**Relationships**:
- Read by linkedin-post workflow to determine posting schedule
- Updated by operator via n8n UI

### Entity: Feed Discovery Result

A relevant LinkedIn post found during browser-based feed browsing.

**Attributes**:
- `urn`: LinkedIn post URN (unique identifier)
- `author_name`: Post author display name
- `author_headline`: Author's LinkedIn headline
- `content_snippet`: First ~200 characters of post text
- `post_url`: Direct URL to the LinkedIn post
- `discovered_at`: Timestamp of discovery
- `topics_matched`: List of matched topic keywords
- `suggested_comment`: AI-generated comment suggestion (may be null)
- `operator_action`: `pending` | `commented` | `liked` | `skipped` | `expired`

**Persistence**: Stored in n8n Workflow Static Data during discovery session. Presented to operator via chat. Expired after 24 hours (LinkedIn feed content becomes stale quickly).

**Relationships**:
- May produce one or more Content Drafts (comment, like)
- Produces Activity Records for discovery sessions

### Entity: Browser Session State

The LinkedIn browser session used by Playwright for feed discovery.

**Attributes**:
- `storage_state_path`: Path to the storageState JSON file (Docker volume mount)
- `session_created_at`: Timestamp of initial manual login
- `last_verified_at`: Timestamp of last successful health check
- `status`: `valid` | `expired` | `unknown`
- `alert_sent`: Whether the session expiry alert has been delivered

**Persistence**: storageState JSON file stored on Docker volume, surviving container restarts. Health check status tracked in n8n Workflow Static Data.

**Session lifecycle**:
1. Operator performs manual login in headed browser (initial setup)
2. storageState exported and mounted into Docker volume
3. Each discovery session verifies session health before browsing
4. If session expires, alert sent to operator for manual re-login
5. LinkedIn `li_at` cookie officially expires after 1 year; real-world sessions last weeks to months
6. Sessions from headless/incognito browsers expire within ~1 hour (must use headed, non-incognito for initial login)

**Credential isolation**: Browser session lives inside n8n Docker container. OpenClaw never accesses the storageState file or browser cookies.

**Relationships**:
- Required by feed-discovery workflow (Phase 2)
- Monitored by browser session health check workflow (Phase 3)
- Alert produces Activity Records

### Entity: Scheduled Action

An engagement action (like, comment) approved by the operator and queued for future execution at a specific time.

**Attributes**:
- `id`: Unique identifier (UUID)
- `action_type`: `like` | `comment`
- `target_urn`: LinkedIn post URN
- `content`: Comment text (null for likes)
- `scheduled_at`: Timestamp when this action should be executed
- `status`: `queued` | `executed` | `failed` | `cancelled`
- `created_at`: Timestamp when operator approved
- `executed_at`: Timestamp of actual execution (null until executed)

**Persistence**: Stored in n8n Workflow Static Data, read by the action-runner workflow.

**Scheduling algorithm**: When operator approves a batch of N actions, compute N timestamps spread across the remaining active hours for the day. Use the `timing_randomization_range_minutes` from Operating Configuration to add jitter. Never cluster more than 2 actions within a 30-minute window.

**Relationships**:
- Created when operator approves engagement actions (from Feed Discovery Results)
- Processed by the action-runner scheduled workflow (every 5 minutes)
- Produces Activity Records when executed

### Entity: Extraction Result

Structured facts extracted from LinkedIn feed content by the quarantined extraction agent. This is the sanitized output that the main agent receives — never raw LinkedIn text.

**Attributes**:
- `source_urn`: LinkedIn post URN
- `author`: Post author name
- `topic`: Primary topic classification
- `key_claims`: List of factual claims made in the post (extracted, not quoted)
- `sentiment`: `positive` | `neutral` | `negative` | `mixed`
- `relevance_score`: 0.0-1.0 relevance to configured topics
- `extracted_at`: Timestamp

**Data flow**: Raw LinkedIn content → input sanitization → extraction agent (no tools) → Extraction Result (structured JSON) → main agent (generates comment suggestion). Raw content never reaches the main agent.

**Relationships**:
- Produced by the extraction agent from Feed Discovery Results
- Consumed by the main agent for comment suggestion generation
- Referenced by Content Drafts (comments generated from this data)
